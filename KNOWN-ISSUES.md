# Known Issues

> Surfaced by the July 2026 documentation sweep: documenting every user-facing control against the
> source forced a read of the code behind each claim, which turned up defects that normal use hides.
> The sweep's 46 agents reported 233 raw candidates; after deduplication (the worst offenders were
> reported 10+ times each) and discarding non-bugs and stale-comment nits, **every entry below was
> individually re-verified against the source**, and the three compiler claims were additionally
> reproduced by running the shipped compiler in Node. Confidence caveats are inline where a claim
> rests on reading draw paths rather than observing pixels.
>
> Not listed here because they are already documented where they matter: the WASM evaluator hang
> (see the [hang dossier in the developer docs](https://docs.rmt.world/developer/wasm/overview#status-the-evaluator-hot-swap-is-blocked)),
> the ~3.5 s sample cap on held notes (instruments guide), and `sync()` cost at 100k notes
> (performance deep-dive).

## Broken user-facing behaviour

- [ ] **Ctrl/Cmd+Shift+Z performs UNDO.** The keydown handler (player.js:6549-6552) lowercases `e.key` and never checks `e.shiftKey`, so the near-universal redo chord walks the user *backwards* through history — destructive for anyone with that muscle memory. Fix: `if (key === 'z' && e.shiftKey)` → `history:redo` (Ctrl+Y already redoes).
- [ ] **A rejected expression save is completely silent.** The note widget's Save handler wraps validate → simplify → commit in a try/catch whose only action is `console.error` (variable-controls.js:1375-1377). Bad syntax, self-reference and circular deps all throw good, user-grade messages (modals/validation.js:16, 38, 44, 49; `src/dsl/errors.js` even has polished strings) — and every one is discarded. The user presses Save, nothing changes, no red border, no message. Reported 12+ times by the sweep; the single most-hit defect.
- [ ] **Unparseable expressions silently compile to the constant 0 — and every validator passes them.** `compile()` emits `LOAD_CONST 0` with only a console.warn when both parsers fail (expression-compiler.js:97-102), and the legacy `parseAtomic` returns const-0 *without throwing* (:310-312), which also bypasses the DSL retry at :89-92. Verified by running the real compiler: `[1]f` (missing dot), `2 * beat(base)` (helper not at position 0 defeats `isDSLSyntax`, dsl/index.js:41), `new Fraction(5).neg()` (the legacy chain regex has no `neg`) and literal `garbage nonsense` **all compile to constant 0**. Because `validateExpressionSyntax` validates by calling that same never-throwing `compile()` (safe-expression-validator.js:75-86), all four return `{valid: true}`; the widget's legacy path then canonicalizes `[1]f` to `new Fraction(0, 1)` and saves it (modals/validation.js:132-136), and the file-load path checks structure only (player.js:5244-5249). A typo zeroes a note with zero feedback. Fix direction: make `parseAtomic` throw and let `compile()` propagate failure to callers that can show it.
- [ ] **A *valid* DSL expression referencing 17+ distinct notes is silently zeroed.** `BinaryExpression.clone()` sizes the copy's `dependencies` at the default 16 — the constructor arg only sizes `bytecode` — so `copy.dependencies.set(...)` throws `RangeError` at 17+ deps (binary-note.js:305-314 vs the growable `addDependency` at :190-201). `compile()` catches the throw as a "DSL failure", retries the legacy parser, and lands on the constant-0 fallback. Reproduced: `[1].f + [2].f + … + [17].f` → bytecode `LOAD_CONST 0`, no error anywhere. One-line fix: size the clone's array from `this.depCount`.
- [ ] **"Copy to Modules" silently drops every note's color and instrument.** `buildModuleFromSelection` reads `n.color` / `n.instrument` / `base.instrument` (player.js:4330-4331, 4349), but `Note` keeps these in `this.properties` (note.js:34-37) and exposes no such getters — the guards are always undefined. The module's own export path proves the correct access (`note.properties.color`, module.js:927-928). Fix: read `n.properties.*`.
- [ ] **The theme dropdown can silently revert.** Selecting a preset first writes `appearance.overrides = {}` (settings-panel.js:259), which synchronously fires `syncControls()`, whose registered syncer re-seeds the `<select>` back to the OLD `themeId` (:264) — before :260 reads `presetSel.value`. Interactive use is saved only by the focus guard (:140, skips `document.activeElement`); any `change` that lands while the select is unfocused silently reverts. One-line fix: capture `presetSel.value` into a local before the `overrides` write.
- [ ] **Arrows tab: "Independent up/down" is offered but unusable, and bad ratios are silently replaced by the octave.** The mode dropdown offers independent (settings-panel.js:304) and the schema honours a stored independent `down` (settings-schema.js:162-168), but the tab renders a ratio editor for `arrows.up` only (:307-344) — selecting Independent strands `down` at whatever it last held with no UI to change it. Separately, an out-of-range ratio (`17/1`, `0/1`, `3/3`) falls back to the SCHEMA DEFAULT `2/1`, not the user's previous value — `validateRatio` (settings-schema.js:115-127) is re-run over the whole tree with fresh defaults on every write (settings-store.js:120-127) — and the panel re-seeds the inputs, so a carefully-set interval visibly rewrites itself to 2/1 with no message.
- [ ] **Transport: a dead window after pause, and loop mode desyncing from its icon.** (a) Clicking Play during the pause fade does nothing: `pause()` leaves `isPlaying` true until the fade promise resolves (player.js:4480-4486), so the click takes the `if (isPlaying) pause()` branch (:4960) and `pause()` early-returns on `isPaused` (:4457). (b) A background tap while looping calls `stop(false)` (:1385), which deliberately preserves `isLoopEnabled` (:4498-4500) — but `.looping` is only applied while playing (:4546), so the transport looks stopped and non-looping while the next Play unexpectedly loops. (c) When the engine *refuses* a loop, `play()` zeroes `loopPeriod` (:4444) but leaves `isLoopEnabled` set, so the icon orbits while playback won't actually loop. Also: the comments at :4488-4489 and :4539-4541 both claim pause keeps the mode armed; the code at :4475 clears it.
- [ ] **Selection, hover and group rings — and all on-note text — are hardcoded white; on Mono Light they approach white-on-white.** Selection ring `gl.uniform4f(uCol, 1,1,1,1)` (renderer.js:2913), selection wash (:2870), hover ring (:2988), group ring (`_multiSelRingRgba`, :1521-1523), and the glyph rasterizer bakes `#ffffff` (:7926, :8145, :8437 — fractions, "silence", ▲/▼). The shipped Mono Light preset has bg `#f5f5f0` (presets.js:86) and defines `hoverRing: #333333` / `selectionRing: #d17400` tokens that never reach these draws. Severity judged from the draw paths, not a screenshot — but white feedback on a near-white canvas can't be right.
- [ ] **The note widget misreports state.** (a) BaseNote instrument row: with no pinned instrument, `collectVariables` hardcodes `evaluated: 'sine-wave', isInherited: false` (modals/index.js:282-289) instead of `module.findInstrument(note)` — set `audio.defaultInstrument` to piano and the widget says `Current: sine-wave` while playback uses piano. (b) The per-property `isCorrupted` flag is computed and passed for startTime/duration/frequency (modals/index.js:296, 313, 322) and **never read**: `buildEvaluatedDiv` (variable-controls.js:48-78) consults only `isTransitivelyCorrupted` (set for frequency only, index.js:324) and `_irrational`/`_floatValue` (set *only* by the WASM adapter, evaluator-adapter.js:908-909, which never runs). An irrational duration or startTime displays as a plain fraction with no `≈`.
- [ ] **Loading a non-JSON module file fails with no message.** `loadModule` does `JSON.parse` inside `file.text().then(...)` and only the inner `Module.loadFromJSON` promise has a `.catch` — the outer chain has none (player.js:5241-5283), so a parse throw is an unhandled rejection: no notify, no error, nothing. One-line fix: add a `.catch` on the outer chain.

## Controls that do nothing

- [ ] **Six of the fifteen Appearance color pickers don't do what they say; five do nothing at all.** Grep-verified: the only `_themeColors.*` reads in renderer.js are `accent/accentHex/noteBorder/measureBar` (:378-381) and `selectionRing` (:10833). Inert: **Hover ring** (`tc.hoverRing` stored at :362, never read — ring is white, :2988), **Frequency / Start time / Duration** dependency-highlight pickers (`tc.depFrequency/depStartTime/depDuration` stored :363-365, never read — the rings draw from hardcoded literals), and **Panel surface** (`--rmt-surface` written by theme-manager.js:54, declared styles.css:14, zero `var(--rmt-surface)` consumers anywhere — every match is `--rmt-surface-border`; panels actually use `rgba(var(--rmt-bg-rgb), 0.88)`). Mislabeled: **Selection ring** only tints the multi-select *marquee rectangle* (:10833); the ring around a selected note is hardcoded white (:2913). Either wire these through or drop the pickers.
- [ ] **The sample manifest's `envelope` block is parsed and ignored.** `_loadManifest` merges `m.envelope` into `this.envelope` (multisample-instrument.js:60), but `getEnvelopeSettings()` returns hardcoded ratios (:183-190) and `applyEnvelope` consults only `this.gain` (:192-195). Editing `envelope` in `public/samples/*/manifest.json` changes nothing; only `gainDb` works.
- [ ] **Theme preset geometry and three tokens can never apply.** `resolve()` falls back to `preset.geometry.*` only via `??` (theme-manager.js:107-111), but `validateSettings` always fills `appearance.note.*` with clamped numbers (settings-schema.js:156-158), so the fallback never fires — High Contrast's declared `borderPxAtZoom1: 2, roundedCornerPxAtZoom1: 4` (presets.js:129) silently renders as 1/6. `noteDefaultSaturation` and `newNoteColorMode` are declared in all four presets (presets.js:43-45 etc.) and read by nothing; `accentText` → `--rmt-accent-text` (theme-manager.js:52) has zero consumers and no picker.

## Misleading copy

- [ ] **Clean Slate claims "This action cannot be undone."** It can: `cleanSlate()` ends with `captureSnapshot('Clean Slate')` (player.js:581) and Ctrl/Cmd+Z restores everything. Dialog text at modals/index.js:566. Reword — copy that lies about undo teaches users not to trust the copy.
- [ ] **Group delete claims the action is "irreversible."** `deleteMultiSelection` captures a single `Delete N Notes` snapshot (player.js:793); one undo restores the lot. Copy at group-widget.js:152.
- [ ] **The delete dialog promises "notes highlighted in red" — nothing turns red.** `showDeleteConfirmation` promises it (modals/index.js:473-474); `highlightDependencies` (:219-249, called at :148) adds `.dependency`/`.dependent` classes to `.note-content` elements the GL renderer never creates — and the CSS for those classes was itself removed (`/* removed legacy */`, styles.css:1422-1430), so even the still-DOM measure triangles and base circle get an unstyled class. The function is effectively dead; the dialog copy promises its ghost.

## Latent / dead code

- [ ] **The evaluator degrades silently instead of erroring — three flavours.** (a) The loader accepts note ids up to 100000 (module.js:852) but `LOAD_REF` encodes ids as u16 (binary-note.js:117-121): `[70000].f` silently binds to note 4464. (b) `removeNote()` never rewrites or flags dependents (module.js:568-578); a dangling `LOAD_REF` pushes hardcoded defaults — frequency 440, startTime 0, duration 1 — with no warning at all (binary-evaluator.js:965-977, same for `LOAD_BASE` at :1008-1018). The UI's delete paths liberate dependents first, so today this traps programmatic callers and hand-authored JSON. (c) Division by zero pushes 1 with only a console.warn (:1049-1061) — `base.f / 0` evaluates to 1.
- [ ] **~400 lines of dead widget UI, one of them leaking listeners.** `createMeasureBeatsSelector` (variable-controls.js:556-885) — the full measure editor with note-length icons, dot modifiers and a beats input — is never exported or called; `modals/index.js:160` uses the bare `createMeasureDurationRow` instead. The custom dark instrument dropdown (variable-controls.js:1042-1110) is built, never appended — :1176 moves the native `<select>` out of the orphaned wrapper — **and registers a document-level capture click listener per widget build (:1107) that is never removed.** Also dead: `src/module-serializer.js` (zero imports repo-wide) and the `camera.shouldAllowSingleFingerPanStart` hook (workspace.js:123 — the assignment is the only reference; CameraController never calls it).
- [ ] **384 KB of WASM is fetched and instantiated on every page load, then never used.** `src/wasm/index.js` self-runs `initWasm()` on import (:125-134), but `isEvaluatorHotSwapEnabled()` is false in a browser without `?evaluator=wasm` (evaluator-adapter.js:36-40) — so the default path pays the download+instantiate for nothing (rmt_core_bg.wasm = 384,008 bytes). Three of the `WASM_CONFIG` flags (`useFractions`/`useGraph`/`useCompiler`, wasm/config.js:11-14) gate adapters that nothing imports — dead switches. Either lazy-init behind the gate or accept the cost knowingly.
- [ ] **The GPU-picking scaffold allocates a full-canvas FBO it never draws into.** `Workspace.init()` constructs and inits `Picking` on every boot (workspace.js:107-111), allocating a canvas-sized RGBA8 texture + depth renderbuffer — but `Picking.begin()`/`end()` have zero call sites and the file's own header says "scaffold, no draw integration yet" (picking.js:1-9). CPU picking is the real path. Stop allocating it, or finish it.
- [ ] **Legacy-syntax still leaks into DSL modules.** The base note's `measureLength` class default is a legacy method-chain string (module.js:87) and `createModuleJSON` exports every base expression with a source (:904-910), so **every saved module gains a legacy `measureLength` even when the source file was pure DSL**. The measure-duration row's fallbacks write `new Fraction(4)` (variable-controls.js:911, 934). And octave arrows on a legacy `.pow(` frequency wrap `new Fraction(n,d).mul(<raw>)` per press instead of folding (player.js:3740-3744) — the value stays correct but the expression nests multipliers without bound; the DSL branch folds properly.
- [ ] **Every Playwright perf harness defaults to a port nothing serves.** The dev server is pinned to 3000 (vite.config.js:5), but bench-render.mjs:29 defaults to `:5173` and bench-drag.mjs:14, bench-hover.mjs:10, bench-pick.mjs:10, converge.mjs:17, profile-sync.mjs:12, visual-regress.mjs:26, who-dirties.mjs:10 (plus drag-shot/shot-arrows/shot-heights) default to `:3001` — while most `shot-*` scripts correctly default to `:3000`. Three inconsistent defaults; running any bench without `--url` connects to a dead port.
- [ ] **`scripts/gen-melodies.mjs` is behind the shipped content — re-running it regresses three melodies.** Commit 30d8a64 hand-edited `amazing-grace.json`, `bach-minuet.json` and `greensleeves.json` without touching the generator (`git log -- scripts/gen-melodies.mjs` confirms). Anyone running the generator clobbers the corrections. Back-port the fixes into the generator or mark those three as hand-maintained.
- [ ] **Declarations that contradict the code.** settings-schema.js:7-8 still asserts "reverb and stereo default OFF" while `defaultSettings()` sets `reverb.enabled: true` (:72). settings-panel.js:8-10 claims a main-menu "Settings…" entry (index.html has only `#settingsGearBtn`) and "Four tabs" (TABS at :87-93 defines five — Scale was added). index.html:141 ships the lock button as `class="lock-button locked" aria-pressed="true"` while player.js:38 boots `isLocked = false` — `updateLockButton()` corrects the DOM at init, but until then the padlock renders closed and misreports to assistive tech.
